## Pre-LLM Calibration Rule

This is a hard-coded logic gate.

The LLM does not decide this rule.

Backend rule:

IF any axis in Pass 2 has `score: 1`
OR the `Evidence` axis score is less than or equal to 2:

The decision must be one of:
- "PARK"
- "PIVOT"

IF all axes are greater than or equal to 3
AND `Pain` is greater than or equal to 4:

The decision may be:
- "PROCEED TO PROTOTYPE"

Otherwise:

The decision should be:
- "INVESTIGATE"

Decision meanings:

- "PARK" = stop active building for now; evidence is too weak
- "PIVOT" = revise the target user, problem, wedge, or trust model before continuing
- "INVESTIGATE" = promising but not prototype-ready; run discovery experiments
- "PROCEED TO PROTOTYPE" = evidence is strong enough for a narrow prototype

## Decision Rule

Set `decision` equal to the backend-provided `calibration_decision`.

Do not soften the decision.

Do not write:
- "PARK for now, but proceed..."
- "PIVOT, but build anyway..."
- "INVESTIGATE, but prototype..."

If the decision is "PARK", the brief should make clear that active building should stop until evidence improves.

If the decision is "PIVOT", the brief should make clear what must change before continuing.

If the decision is "INVESTIGATE", the brief should focus on discovery tasks, not implementation.

If the decision is "PROCEED TO PROTOTYPE", the brief should still recommend a narrow prototype, not a full product.

## Field Rules

### one_sentence_idea

Write one sentence.

Base it on the raw idea and supported assumptions.

Do not add new product capabilities.

Good:
"Build a secure shared memory layer that helps AI tools preserve project context across apps."

Bad:
"Build the de facto personal AI operating system for every agent."

### target_user

Describe only the user evidence that exists.

If the target user is broad or missing, say so plainly.

Good:
"Not yet specific; the transcript points to multi-tool AI builders but names no real users."

Bad:
"AI-native professionals and enterprise teams everywhere."

### pain_chain

Describe the sequence:

trigger -> user behavior -> cost or consequence

If cost or consequence is missing, say it is not yet quantified.

Good:
"Agents lose context, users repeat project background, and the cost is not yet quantified."

### current_workaround

Describe current alternatives or workarounds from the assumptions.

If none are validated, say:
"Current workaround not yet validated."

Do not invent likely workarounds unless the assumption map includes them.

### riskiest_assumption

Choose the highest-risk critical assumption.

Prefer assumptions that are:
- critical
- low evidence
- tied to Pain, User, Technical, or Evidence scores

Use the assumption label, not a long paragraph.

### evidence_needed

State the minimum evidence required to move the decision forward.

Make it concrete.

Good:
"Observed examples from 5 target users showing repeated context-loss rework."

### invalidation_threshold

Define what result would kill or force revision of the idea.

It must be measurable.

Good:
"If fewer than 3 of 10 target users report weekly context-loss pain, park the idea."

Bad:
"If users do not like it."

### next_3_experiments

Return exactly 3 experiments.

Each experiment must:
- test a critical or low-scoring assumption
- be doable before building a full product
- have a measurable success signal
- include a short timebox

For "PARK":
- experiments should be evidence recovery tasks
- do not suggest prototype building unless the missing evidence is technical only

For "PIVOT":
- experiments should test the revised user/problem/wedge

For "INVESTIGATE":
- experiments should combine discovery and one narrow feasibility test if useful

For "PROCEED TO PROTOTYPE":
- experiments may include a narrow prototype
- still avoid full build plans

### confidence_level

Use:

- "low" when Evidence score <= 2 or any axis is 1
- "medium" when most axes are 3 and no blocking score exists
- "high" only when Pain >= 4, Evidence >= 4, and no axis is below 3

### honest_gaps

List 3–6 gaps.

Each gap should be short and specific.

Good:
- "No named first users yet."
- "No observed context-loss incidents yet."
- "Trust boundary is untested."

Bad:
- "Need more research."

## PARK vs PIVOT Guidance

When the backend gate requires "PARK" or "PIVOT", choose the backend-provided `calibration_decision`.

If the backend has not provided a specific blocked decision, use this logic:

Choose "PARK" when:
- evidence is too weak across the board
- the idea may be plausible but user pain is not validated
- active building would mainly create false confidence

Choose "PIVOT" when:
- there is some evidence, but it points to a different user, problem, wedge, or trust model
- the broad idea is too large but a narrower validated wedge may exist
- the strongest assumption contradicts the current framing
